## 🛠️ Important Architectural Decisions
*(AI: Never delete these entries. Add to them when making technical choices.)*

**Decision:** Use an environment variable (`APP_MODE`) to switch between "production" and "demo" modes.
**Why:** This is the safest and most explicit way to control the application's behavior. It prevents developers from accidentally committing a "demo" configuration to production and makes activating demo mode a simple, standard process.

**Decision:** Isolate all demo-specific assets in a dedicated `scripts/demo/` folder.
**Why:** This creates a clear separation between live application data (like the production compliance Excel file) and temporary, non-critical demo assets. It improves clarity and reduces the risk of accidental data crossover.

**Decision:** Make setup scripts idempotent.
**Why:** The `seed_demo_database.py` script now deletes the old database before running. This ensures that the demo environment is always reset to a clean, known state, no matter how many times the script is run, leading to predictable and reliable demonstrations.
