## 🛠️ Important Architectural Decisions
*(AI: Never delete these entries. Add to them when making technical choices.)*

**Decision:** Use a central Excel file on a shared network drive as the single source of truth for compliance data.
**Why:** This model provides the ultimate simplicity for a non-technical user. It avoids the complexity and brittleness of web scrapers and the usability hurdles of Git or Google Sheets API authentication. The workflow is familiar (editing a spreadsheet) and safe (isolated from the core application).

**Decision:** Use two separate, scheduled scripts for "change detection" and "data ingestion."
**Why:** This decouples the risky, external-facing task (checking websites) from the critical internal task (updating the database). The weekly "watchdog" script can fail without impacting the nightly sync of existing, verified data.

**Decision:** Implement a "read-modify-write" cycle for all Excel operations.
**Why:** This ensures that updates to the Excel file are atomic. It prevents partial writes and data corruption, guaranteeing that user feedback (like the `Sync Status`) is always saved correctly.

**Decision:** Fail loudly on configuration errors.
**Why:** The automated scripts will now immediately send an email alert if the configured Excel file is missing or inaccessible. This prevents silent failures and ensures that operational issues are reported to the technical owner instantly.
